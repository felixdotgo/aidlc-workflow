
# AI-DLC · Index

Generate two small Markdown manifests so later phases read a tiny index instead of scanning the whole tree. This is the main token lever of the workflow.

**Outputs (overwrite each run):**
- `.agents/state/repo-map.md` — submodule → stack → service → URL → test/lint command.
- `.agents/state/specs-index.md` — every spec/doc → area → submodule → keywords → 1-line summary.

Both are pure Markdown. No scripts, no Python. For a large tree you MAY delegate the scan to one `Explore` subagent and write the files yourself from its summary.


## Step 1 — repo-map.md

1. Read `.gitmodules` for the submodule list.
2. For each submodule, detect the stack from marker files (common examples — extend per project): `composer.json` (+ `artisan`) → **Laravel** (read the language version from the manifest); `package.json` (framework dep, e.g. `nuxt`/`next`/`vue`) → that JS framework; `requirements.txt` / `pyproject.toml` (+ framework import, e.g. FastAPI/Django) → that Python framework. Match each detected stack to its `rules/<stack>-conventions.md` (this dir) if one exists.
3. From the **root** compose file: map each submodule to its service name(s), profile/selector, and the reverse-proxy host rule (e.g. Traefik `Host(...)`) → URL. Submodules absent from root compose run standalone — note that.
4. Derive the narrowest test/lint command per stack (see table below).
5. **Build the feature-area / screen-code map.** For each spec repo under `spec/` (and any spec dir the user names), read ~30 lines of its `INDEX.md` / `Function_list.md` (or any screen listing) to detect the project's screen-code prefix convention `<AREA>_<NN>` and any ID-infix convention. Pair each prefix with its backend submodule by name (match the `<area>_*` ↔ `<area>_spec_ba` naming) and the spec repo path. For a backend whose spec repo has **no real content yet**, still emit a row with its assumed prefix marked `(TBD)` so the structure is visible. Also list non-screen areas by app name/alias (prefix `—`).
6. Write `.agents/state/repo-map.md` using this exact shape:

```markdown
# Repo Map (AI-DLC) — generated by aidlc-index

| submodule | stack | service(s) | profile | URL | narrowest test / lint |
|-----------|-------|-----------|---------|-----|-----------------------|
| <submodule> | <stack + version> | <service(s)> | <profile / selector> | <URL> | <narrowest test cmd> · <lint cmd> |

## Feature areas / screen-code map
> Resolve a screen code or app name in a request → backend + spec. Pattern: `<AREA>_<NN>`; an ID infix `*-<AREA>-*` ⇒ same area. `(TBD)` = prefix/spec assumed, not yet confirmed.

| screen prefix | feature area (app) | aliases | backend submodule | spec repo |
|---------------|--------------------|---------|-------------------|-----------|
| `<AREA>_*` | <feature area> | <aliases> | <backend submodule> | <spec repo path> |

## Shared infra
<list the shared services from the root compose: db, cache/queue, object store, reverse proxy — with ports/URLs and how profiles/services are selected>.

## Standalone (NOT in root docker-compose.yml)
- <submodule> — <how it runs / own compose> — <test cmd>
```

**Deriving the narrowest test/lint command:** assume the stack is already running (lifecycle principle: `conventions.md §7`). For each submodule, derive the tightest single-test + lint command for its detected stack from the project's own docs/config — the submodule's `CLAUDE.md`, its test runner config, and the project's `rules/stack-runtime.md` (concrete project commands). Record the resolved per-submodule commands in the table above; don't hardcode stack commands into this phase doc.


## Step 2 — specs-index.md

1. Determine spec roots, in order (discover layout from `.gitmodules` + the project's CLAUDE.md — don't assume fixed dir names):
   - The project's **spec submodules** (if any) — typically one repo per feature area, each its own git repo. Index every `*.md` / `*.mmd` inside.
   - Any other top-level dir the user names as a spec repo.
   - Internal docs (always include): per code-submodule `docs/**/*.{md,mmd}`, `swagger.{yml,yaml}` / `openapi.{yml,yaml}`, and top-level + per-submodule `README.md`.
   Discover with `find` (nested spec repos contain `.git`, so prefer `find` over `rg` here), and do not open whole files — read only the first ~30 lines for the title/keywords. Skip template-only READMEs but list them as `⚠️ placeholder`.
2. For each file, capture: H1 title (or filename), a 1-line summary, 3–6 keywords, and the submodule/area it concerns. Read only the first ~30 lines of each to extract this — never the whole file.
3. Write `.agents/state/specs-index.md`:

```markdown
# Spec Index (AI-DLC) — generated by aidlc-index
Read THIS first to locate specs; then open ONLY the matched files.

Spec roots scanned: <list>. Spec submodules: <present | ABSENT>.
> If a spec repo is still a template README, list it as `⚠️ placeholder`; re-run after it gets real content.

| doc | area | submodule | keywords | summary |
|-----|------|-----------|----------|---------|
| <path/to/doc> | <area> | <submodule> | <3–6 keywords> | <1-line summary> |
```


## Step 3 — finish

- Report what changed (counts: N submodules, M spec files indexed) and any submodule whose stack could not be detected.
- Do **not** create an AI-DLC task or touch `BOARD.md` — indexing is maintenance, not a task.
